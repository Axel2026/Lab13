TEMAT : Zapytania wybierające z klauzulą WHERE

  1. Wprowadzenie

    1. Definiowanie zapytania wybierającego z klauzulą WHERE.

      Klauzula WHERE jest klauzulą selekcji, która wybiera z tabeli wszystkie rekordy spełniające określony warunek. Przy definiowaniu kryteriów wyboru rekordów z tabeli można wykorzystać następujące klasy predykatów:

      Klasa predykatów Alternatywna notacja
      		 Opis
      =, <>, >, >=, <, <=
      		Porównuje dwie wartości, dwa wyrażenia
      BETWEEN...AND...
      		Sprawdza czy wartość znajduje się w określonym zakresie (przedziale domkniętym)
      IN(...,...)
      		Sprawdza czy wartość znajduje się na zadanej liście (w zadanym zbiorze)
      LIKE
      NOT LIKE
      		 ~~
      !~~
      		 Sprawdza czy wartość odpowiada podanemu wzorcowi
      (% - oznacza dowolny łańcuch znaków /także zerowy/,
      _ - oznacza dowolny, dokładnie jeden znak)
      ILIKE
      NOT ILIKE
      		 ~~*
      !~~*
      		 j.w. (nie jest brana pod uwagę wielkość liter)
      SIMILAR TO
      NOT SIMILAR TO
      		Sprawdza czy wartość odpowiada podanemu wzorcowi
      (% - oznacza dowolny łańcuch znaków /także zerowy/,
      _ - oznacza dowolny, dokładnie jeden znak,
      | - oznacza alternatywę,
      * - oznacza powtórzenie poprzedniego znaku 0 lub więcej razy,
      + - oznacza powtórzenie poprzedniego znaku 1 lub więcej razy)
      W odróżnieniu od predykatu LIKE/ILIKE może wykorzystywać wyrażenia regularne
      ~
      !~
      		Sprawdza czy wartość odpowiada podanemu wzorcowi
      (. - oznacza dowolny znak,
      | - oznacza alternatywę,
      * - oznacza powtórzenie poprzedniego znaku 0 lub więcej razy,
      + - oznacza powtórzenie poprzedniego znaku 1 lub więcej razy)
      W odróżnieniu od predykatu SIMILAR TO może wykorzystywać wyrażenia regularne standardu POSIX
      ~*
      !~*
      		j.w. (nie jest brana pod uwagę wielkość liter)
      IS NULL
      IS NOT NULL
      		Sprawdza czy wartość jest wartością NULL
      Sprawdza czy wartość nie jest wartością NULL

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_tabeli

      		WHERE warunek;

      Przykłady:

      a) Operatory LIKE (~~), SIMILAR TO oraz wyrażenia regularne POSIX.

      b) Operatory BETWEN .. AND, IN, IS oraz OVERLAPS.



    2. Definiowanie zapytania wybierającego z klauzulą WHERE oraz operatorami logicznymi.

      3. Klauzula WHERE może zawierać operatory logiczne OR, AND oraz NOT.

      A OR B TRUE FALSE NULL
      TRUE TRUE TRUE TRUE
      FALSE TRUE FALSE NULL
      NULL TRUE NULL NULL

      A AND B TRUE FALSE NULL
      TRUE TRUE FALSE NULL
      FALSE FALSE FALSE FALSE
      NULL NULL FALSE NULL

      A NOT A
      TRUE FALSE
      FALSE TRUE
      NULL NULL

      Przykłady:

      SELECT *

      		FROM dziekanat.studenci

      		WHERE imie LIKE '%a' AND nazwisko LIKE 'A%';

      SELECT *

      		FROM dziekanat.przedmioty

      		WHERE nazwa_przedmiotu = 'Algebra liniowa' AND nr_semestru = 2;



    3. Definiowanie zapytania wybierającego z klauzulą ORDER BY.

      Klauzula ORDER BY określa kolejność rekordów w jakiej zostaną zwrócone jako rezultat zapytania. Rekordy wynikowe mogą być posortowane w następujący sposób:

      • rosnąco ASC (ang. ascending);
      • malejąco DESC (ang. descending).

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_tabeli

      		ORDER BY nazwa_kolumny [ASC | DESC] [NULLS {FIRST | LAST}],...,nazwa_kolumny [ASC | DESC] [NULLS {FIRST | LAST}];

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_tabeli

      		ORDER BY nr_kolumny [ASC | DESC] [NULLS {FIRST | LAST}],...,nr_kolumny [ASC | DESC] [NULLS {FIRST | LAST}];

      Przykłady:

      SELECT *

      		FROM dziekanat.studenci

      		ORDER BY nazwisko, imie ASC;

      SELECT *

      		FROM dziekanat.studenci

      		ORDER BY 3, 2 ASC;

      SELECT nr_albumu, imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY 1 ASC;

      SELECT nr_albumu::CHARACTER(5), imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY 1 ASC;

      SELECT lpad(nr_albumu, 5, '0')::CHARACTER(5), imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY 1 ASC;

      SELECT nr_albumu, imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY lpad(nr_albumu, 5, '0')::CHARACTER(5) ASC;

      SELECT nr_albumu::INTEGER, imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY 1 ASC;

      SELECT nr_albumu, imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY nr_albumu::INTEGER ASC;



    4. Definiowanie zapytania wybierającego z klauzulami ALL oraz DISTINCT.

      Klauzula DISTINCT stosowana jest w celu wyeliminowania powtarzających się rekordów.

      Składnia:

      SELECT [ALL | DISTINCT [ON klauzula_nazwa_kolumny_1,..., klauzula_nazwa_kolumny_m]] [nazwa_kolumny_1 AS alias_kolumny_1,...,nazwa_kolumny_n AS alias_kolumny_n | *]

      		FROM nazwa_tabeli;

      Przykłady:

      SELECT DISTINCT wku AS "WKU"

      		FROM dziekanat.wojsko;

      SELECT ALL wku AS "WKU"

      		FROM dziekanat.wojsko;

      SELECT DISTINCT tryb_studiow, stopien_studiow

      		FROM dziekanat.kierunki_studiow

      		ORDER BY 1 DESC, 2;

      SELECT DISTINCT ON (tryb_studiow, stopien_studiow) tryb_studiow, stopien_studiow

      		FROM dziekanat.kierunki_studiow

      		ORDER BY 1 DESC, 2;

      SELECT DISTINCT ON (tryb_studiow) tryb_studiow, stopien_studiow

      		FROM dziekanat.kierunki_studiow

      		ORDER BY 1 DESC, 2;

      SELECT DISTINCT ON (tryb_studiow, stopien_studiow) tryb_studiow

      		FROM dziekanat.kierunki_studiow

      		ORDER BY 1 DESC;

      .:: Intencjonalne pomijanie danych przy użyciu klauzuli DISTINCT ON ::.

      W przypadku użycia klauzuli DISTINCT ON może nastąpić pominięcie określonych danych.

      Przykład:
      1) Poniższe zapytanie zwraca wynagrodzenia wszystkich prowadzacych posortowane rosnąco wg stanowiska oraz malejąco wg płacy zasadniczej z wartościami NULL umieszczonymi na końcu. Rezultat może zawierać duplikaty.
      SELECT stanowisko, placa_zasadnicza
      FROM kadry.prowadzacy
      ORDER BY 1, 2 DESC NULLS LAST;

      2) Stosując klauzulę DISTINCT ON możliwe jest wybranie spośród wszystkich rekordów zwróconych przez w/w zapytanie (1), tylko tych, które znajdują się na początku każdej grupy stanowisk. W rezultacie zapytanie zwraca najwyższe wynagrodzenia na poszczególnych stanowiskach.
      SELECT DISTINCT ON (stanowisko) stanowisko, placa_zasadnicza
      FROM kadry.prowadzacy
      ORDER BY 1, 2 DESC NULLS LAST;

TEMAT : Zapytania z operatorami UNION, INTERSECT, EXCEPT

  1. Wprowadzenie

    1. Definiowanie łączenia tabel z wykorzystaniem operatora UNION.

      Klauzula CORRESPONDING BY nie jest dla operatora UNION wspierana przez PostgreSQL.

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_schematu_1.nazwa_tabeli_1
      UNION [ALL | CORRESPONDING [BY [nazwa_kolumny,...,nazwa_kolumny]]]
      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_schematu_2.nazwa_tabeli_2

      Przykład:

      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 1
      UNION
      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 5;

      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 1
      UNION ALL
      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 5;


    2. Definiowanie łączenia tabel z wykorzystaniem operatora INTERSECT.

      Operator INTERSECT realizuje przecięcie dwóch tabel, którego rezultatem jest tabela wynikowa zawierająca wszystkie rekordy, które są wspólne dla obu tabel.

      Klauzula CORRESPONDING BY nie jest dla operatora INTERSECT wspierana przez PostgreSQL.

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_schematu_1.nazwa_tabeli_1
      INTERSECT [ALL | CORRESPONDING [BY [nazwa_kolumny,...,nazwa_kolumny]]]
      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_schematu_2.nazwa_tabeli_2

      Przykład:

      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 1
      INTERSECT
      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 5;


    3. Definiowanie łączenia tabel z wykorzystaniem operatora EXCEPT.

      Operator EXCEPT realizuje różnicę dwóch tabel, której rezultatem jest tabela wynikowa zawierająca wszystkie rekordy tabeli pierwszej nie występujące w tabeli drugiej.

      Klauzula CORRESPONDING BY nie jest dla operatora EXCEPT wspierana przez PostgreSQL.

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_tabeli_1
      EXCEPT [ALL | CORRESPONDING [BY [nazwa_kolumny,...,nazwa_kolumny]]]
      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_tabeli_1

      Przykład:

      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 1
      EXCEPT
      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 5;

    1. Wprowadzanie pojedynczego rekordu:

      Składnia:

      INSERT INTO nazwa_tabeli [(nazwa_kolumny_1,...,nazwa_kolumny_n)]

      		VALUES (wartosc_1,...,wartosc_n);

      INSERT INTO nazwa_tabeli

      		VALUES (wartosc_1,...,wartosc_n);

      .:: Logiczna niezależność danych ::.

      Zgodnie z #9 postulatem Codd'a dotyczącym logicznej niezależności danych:
      - porządek kolumn w rekordzie tabeli;
      oraz
      - porządek rekordów w tabeli;
      nie mają żadnego wpływu na odczytywanie danych z wykorzystaniem zapytań.
      W związku z powyższym:
      1) kolejność kolumn tabeli, które są uzupełniane danymi jest dowolna (patrz przykład a oraz b) przy warunku, że kolumny są wskazane jawnie*;
      2) liczba kolumn tabeli, które są uzupełnianie danymi jest dowolna (patrz przykład c) przy warunku, że kolumny są wskazane jawnie* oraz przy zachowaniu ograniczeń narzuconych dla tabeli (np. więzów NOT NULL);
      3) pozycja w jakiej rekord zostanie w tabeli dodany jest dowolna (fizyczna lokalizacja rekordu jest nieprzewidywalna i wynika z optymalizacji zapisu danych w pamięci);

      *) W przypadku gdy kolumny nie są wskazane jawnie (patrz przykład d) w SZBD domyślnie przyjmowane jest, że uzupełniane są wszystkie kolumny w kolejności ich występowania w tabeli.

      Przykłady:

      a)

      INSERT INTO dziekanat.studenci (nr_albumu, imie, nazwisko, pesel, data_urodzenia, adres_zamieszkania, adres_korespondencyjny)

      		VALUES ('94875', 'Jolanta', 'Korcz', '86070510220', '1986-07-05', 25, 25);

      b)

      INSERT INTO dziekanat.studenci (adres_korespondencyjny, nazwisko, imie, data_urodzenia, nr_albumu, adres_zamieszkania, pesel)

      		VALUES (26, 'Nowak', 'Andrzej', '1990-02-05', '94876', 26, '90020505575');

      c)

      INSERT INTO dziekanat.studenci (nazwisko, imie, data_urodzenia, nr_albumu, adres_zamieszkania, adres_korespondencyjny)

      		VALUES ('Stern', 'Joanna', '1974-11-02', '94877', 27, 27);

      d)

      INSERT INTO dziekanat.studenci

      		VALUES ('94878', 'Dorota', 'Kowalska', NULL, '1988-12-15', 28, 28);

      .:: Jawne oraz niejawne odwoływanie się do kolumn w poleceniu INSERT ::.

      W rozwiązaniach aplikacyjnych należy mieć świadomość, że struktura bazy danych jest strukturą dynamiczną, która ewoluuje nadążając za zmieniającymi się wymaganiami. W związku z tym istnieje zasadnicza różnica pomiędzy jawnym oraz niejawnym odwoływaniem się do kolumn tabeli.
      Załóżmy, że w tabeli studenci wprowadzone zostają następujące modyfikacje:
      ALTER TABLE dziekanat.studenci ADD COLUMN imie_ojca VARCHAR(25);

      Wykonanie kodu SQL z przykładu 1d (powyżej) w PostgreSQL w wersji poniżej 9.0 zakończy się niepowodzeniem, gdyż liczba dodawanych wartości jest niezgodna (mniejsza) od aktualnej liczby kolumn tabeli studenci. Natomiast kod SQL z przykładów 1a, 1b oraz 1c zostanie wykonany poprawnie.

      *) Aby usunąć kolumnę z tabeli studenci należy wykonać kod SQL:
      ALTER TABLE dziekanat.studenci DROP COLUMN imie_ojca;

      INSERT INTO dziekanat.przedmioty (nazwa_przedmiotu, nr_semestru, ects, id_prowadzacego)

      .:: Kopiowanie danych (np. tabeli) ::.

      W przypadku gdy istnieje konieczność wykonania kopii struktury i/lub danych określonego obiektu (np. tabeli, widoku) można wykorzystać składnię SQL CREATE....AS SELECT lub SELECT INTO.
      Załóżmy, że istnieje potrzeba wykonania kopii tabeli studenci. W przypadku gdy wykonujemy:
      1) kopię wyłącznie struktury obiektu, wykorzystamy kod SQL:
      CREATE TEMPORARY TABLE studenci_kopia AS
      SELECT * FROM dziekanat.studenci WHERE NULL = NULL;
      lub
      SELECT * INTO TEMPORARY TABLE studenci_kopia FROM dziekanat.studenci
      WHERE NULL = NULL;
      2) kopię zarówno struktury jak i wszystkich danych przechowywanych w obiekcie, wykorzystamy kod SQL:
      CREATE TEMPORARY TABLE studenci_kopia AS
      SELECT * FROM dziekanat.studenci;
      lub
      SELECT * INTO TEMPORARY TABLE studenci_kopia FROM dziekanat.studenci;

      *) Klauzula TEMPORARY w definicji tabeli oznacza, że obiekt jest tabelą tymczasową, w związku z tym będzie istnieć tylko w obrębie aktualnej sesji a po zakończeniu zostanie usunięty.



    4. Weryfikacja wprowadzenia zmian tj. dodania rekordów w wykorzystaniem klauzuli RETURNING:

      Składnia:

      INSERT INTO nazwa_tabeli_docelowej [(nazwa_kolumny_1,...,nazwa_kolumny_n)]

      		VALUES (wartosc_1,...,wartosc_n)

      		RETURNING [klauzula_nazwa_kolumny_1,...,klauzula_nazwa_kolumny_m | *];

      INSERT INTO nazwa_tabeli_docelowej [(nazwa_kolumny_1,...,nazwa_kolumny_n)]

      		SELECT [podzapytanie_nazwa_kolumny_1,...,podzapytanie_nazwa_kolumny_n | *]

      		FROM nazwa_tabeli_źródłowej

      		WHERE warunek

      		RETURNING [klauzula_nazwa_kolumny_1,...,klauzula_nazwa_kolumny_m | *];

      Przykłady:

      INSERT INTO dziekanat.przedmioty (nazwa_przedmiotu, nr_semestru, ects, id_prowadzacego)

      		(SELECT nazwa_przedmiotu, 4, ects, id_prowadzacego FROM dziekanat.przedmioty WHERE nr_semestru = 4)

      		RETURNING *;

      UPDATE kadry.prowadzacy

      		SET placa_zasadnicza = 1.1*placa_zasadnicza

      		WHERE stanowisko = 'wykładowca'

      		RETURNING imie, nazwisko, tytul, placa_zasadnicza AS "Podwyższona płaca zasadnicza", (0.1/1.1*placa_zasadnicza)::NUMERIC(6,2) AS "Podwyżka";

      .:: Klauzula RETURNING ::.

      Klauzula RETURNING może być wykorzystywana przy wykonywaniu dowolnej operacji zmiany danych tj. w przypadku:
      - dodawania rekordów poleceniem INSERT
      - usuwania       rekordów poleceniem DELETE
      -       aktualizacji rekordów poleceniem UPDATE

TEMAT : Aktualizacja danych w tabeli

  1. Wprowadzenie
    1. Modyfikowanie wierszy tabeli.

      Składnia:

      .:: Aktualizacje wielokrotne ::.

      Załóżmy, że w wyniku reorganizacji armii, Ministerstwo Obrony Narodowej podjęło decyzję o likwidacji części Wojskowych Komend Uzupełnień, pozostawiając tylko te komendy, które znajdują się w miastach wojewódzkich. Poborowi, należący do likwidowanych WKU, powinni zostać przyporządkowani do właściwych im komend wojewódzkich. Powyższe zadanie może zostać wykonane dwoma drogami:

      A) poprzez wykonanie serii elementarnych aktualizacji z prostymi kryteriami warunkowymi, jednak wymagających wielokrotnego przeglądnięcia danych zawartych w tabeli:
      UPDATE dziekanat.wojsko SET wku = 'WKU w Krakowie'
      WHERE wku IN ('WKU w Tarnowie', 'WKU w Nowym Sączu');
      UPDATE dziekanat.wojsko SET wku = 'WKU w Rzeszowie'
      WHERE wku IN ('WKU w Dębicy', 'WKU w Mielcu');
      UPDATE dziekanat.wojsko SET wku = 'WKU w Kielcach'
      WHERE wku IN ('WKU w Busku Zdroju', 'WKU w Skarżysku Kamiennej');

      B) poprzez wykonanie pojedynczej aktualizacji wymagających jednokrotnego przeglądnięcia danych zawartych w tabeli, jednak ze złożonymi kryteriami warunkowymi:
      UPDATE dziekanat.wojsko
      SET wku = (CASE wku
      WHEN 'WKU w Tarnowie' THEN 'WKU w Krakowie'
      WHEN 'WKU w Nowym Sączu' THEN 'WKU w Krakowie'
      WHEN 'WKU w Dębicy' THEN 'WKU w Rzeszowie'
      WHEN 'WKU w Mielcu' THEN 'WKU w Rzeszowie'
      WHEN 'WKU w Busku Zdroju' THEN 'WKU w Kielcach'
      WHEN 'WKU w Skarżysku Kamiennej' THEN 'WKU w Kielcach'
      ELSE wku
      END)
      RETURNING *;

      UPDATE dziekanat.wku
      SET wku = (CASE
      WHEN wku IN ('WKU w Tarnowie', 'WKU w Nowym Sączu') THEN 'WKU w Krakowie'
      WHEN wku IN ('WKU w Dębicy', 'WKU w Mielcu') THEN 'WKU w Rzeszowie'
      WHEN wku IN ('WKU w Busku Zdroju', 'WKU w Skarżysku Kamiennej') THEN 'WKU w Kielcach'
      ELSE wku
      END)
      RETURNING *;

      Sposób B (pojedyncza aktualizacja) jest bardziej optymalny (koszt czasowy wykonania aktualizacji jest - w zależności od przypadku - od kilkudziesięciu do kilkuset procent mniejszy niż w sposobie A).
